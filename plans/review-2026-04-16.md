## Headline

The pipeline works end-to-end today — the populated
[data/](../data/) tree proves it — and the code is in good shape for a
research tool. Types are strict, tests pass, stages are small and
readable. The three structural gaps that will bite are:

1. **The flag-don't-filter contract is documented but not implemented.**
   `laglitsynth.resolve` does not exist, stage 3 still writes an
   accepted/rejected Work split, and stage 4 copies Works into
   `included.jsonl` rather than writing an `AdjudicationVerdict` sidecar.
   Stage 7 cannot land cleanly until this is resolved.
2. **Reproducibility is the weakest dimension.** Meta sidecars record
   _when_ but not _how_: no git commit, no input paths, no LLM sampling
   parameters, no prompt content digest, no PDF content hash. Replay
   six months from now is impossible.
3. **Terminology drift from the recent rename is pervasive.** The
   rename to two-component stage names (commit `241d33b`) is done at
   the CLI/directory surface but the `Filter*` model names, several doc
   filenames, [docs/pipeline.md](../docs/pipeline.md) section headings
   + mermaid labels, and [README.md](../README.md)'s command examples
   all still carry the old vocabulary.

Everything else is either a straightforward fix or a design call that
can wait until the corresponding stage lands.

## Cross-cutting themes

Each theme groups findings that appeared in at least two sub-reviews.
Ordered by how much downstream work they unblock.

### Theme 1 — Flag-don't-filter cutover

- Surface: stage 3 `--reject-file` and split-files language still
  present in code, doc filenames, and [README.md](../README.md)
  examples; [docs/interfaces.md](../docs/interfaces.md) lines 70–73 /
  152–157 carry a planning statement in what is supposed to be a
  descriptive reference.
- Code: flagged `[blocker]` as the biggest single doc/code drift
  ([`screening_abstracts/screen.py:188–249`](../src/laglitsynth/screening_abstracts/screen.py)).
- System: I-1 (`resolve` missing), I-2 (stage 3 splits), I-3 (stage 4
  copies), I-4 (CLI shape will change) — the whole integration story
  hinges on this.

Why it's theme #1: every downstream stage (5+) was specced on top of
this contract. Adding stage 7 without it means either building stage 7
against the legacy split-files shape and rewriting later, or stalling
stage 7 on a refactor first. Better to land the refactor first, in one
go, per the AGENTS.md "green field, no backwards compatibility"
principle.

### Theme 2 — Reproducibility / provenance

- System: the entire `R-1 … R-20` section. Hard-coded `tool_version =
  "alpha"` in all six meta models; no input paths, no git commit, no
  LLM sampling params, no prompt content digest, no PDF content hash;
  LLM non-determinism not acknowledged in the doc layer; no
  cross-stage join keys.
- Code: `pixi.lock` is gitignored (so a fresh clone may re-solve to
  different dependency versions); no CI; `openai` and `python-dotenv`
  unpinned.

Why theme #2: not urgent for the prototype, but the cost compounds with
every run. The pilot phase is when it's cheapest to bolt on provenance.
Practical start: define a shared `_RunMeta` base with timestamp, tool,
tool_version, tool_commit_sha, input_paths (list), input_content_sha,
counts, and a stage-specific `details` sub-object. Pulling the common
fields together is a one-line change per meta model.

### Theme 3 — Terminology drift after the rename

- Surface: the bulk of the surface review. `Filter*` model names still
  in `screening_abstracts`; `filter_works` function; module docstring
  "LLM-based abstract filtering"; `filtered_at` field; default output
  still uses `_filtered_` in [docs/filter-abstracts.md](../docs/filter-abstracts.md);
  all per-stage docs still use pre-rename filenames
  ([fetch-publications.md](../docs/fetch-publications.md) →
  `catalogue-fetch.md`, [filter-abstracts.md](../docs/filter-abstracts.md)
  → `screening-abstracts.md`, etc.); [docs/pipeline.md](../docs/pipeline.md)
  section headings and mermaid labels still say `screen-abstracts`,
  `full-text-retrieval`, `data-extraction`, `quantitative-synthesis`,
  etc.; [README.md](../README.md) advertises `laglitsynth
  fetch-publications` and `laglitsynth filter-abstracts` (both dead);
  `docs/interfaces.md` has one broken link (`../plans/retuning-propagation.md`
  → should be `../plans/done/retuning-propagation.md`).
- Code: flags the same `Filter*`/`filter_works` naming as renames to
  apply.
- System: D-10 — `AdjudicationMeta.tool` string drift in
  [docs/adjudication-screening.md](../docs/adjudication-screening.md).

Why theme #3: individually these are nits. Together they make the
repo hard to reason about: readers have to hold two vocabularies in
their heads, and every new agent coming to the project pays the
terminology-drift tax.

### Theme 4 — Stage 5/6 small bugs

- Code: `RetrievalStatus.failed` is unreachable, `retrieval.jsonl`
  duplicates on re-run, `_validate_pdf` accepts HTML with
  `Content-Type: application/pdf`, DOI normalisation brittle,
  `_last_request` global state, manual-first cascade differs from the
  doc's OA-first order, stage 6 assumes PDF stems are OpenAlex IDs,
  GROBID preflight reuses the 120 s extraction timeout.
- System: I-5 (stage-6 PDF-stem → work_id without cross-checking
  [retrieval.jsonl](../data/fulltext-retrieval/)), I-6 (unretrieved.txt
  re-scans disk instead of deriving from `RetrievalRecord`).

Why theme #4: each is a 5–15 minute fix, but several of them bake
wrong data into downstream artifacts silently.

### Theme 5 — `Work` / `ExtractedDocument` need fields before late stages

- System: D-1 (no quality_status on `ExtractedDocument`), D-3 (no
  source-catalogue provenance on `Work`), D-4 (no peer-review
  discriminator on `Work`), D-14 (non-nullable topic/keyword fields
  drop whole records silently), D-13 (`ExtractionRecord` shape
  undecided — typed-model vs. `dict[str, FieldValue]`).
- Code: `Work.authorships` / `biblio` have no defaults so sparse
  records are dropped, contradicting "None means None"; `updated_date:
  str` vs `publication_date: date` inconsistency.

Why theme #5: each of these is easier to fix before stage 7+ code
starts reading these models than after. The codebook (stage 8) will
evolve during phase 3 — pin the `ExtractionRecord` design call now.

### Theme 6 — `_Base(extra="ignore")` is too permissive for internal models

- System: D-6. Right for `Work` (OpenAlex may add fields); wrong for
  `FilterVerdict`, `RetrievalRecord`, etc. A typo in a verdict field
  name silently becomes a `None` read downstream, and multi-run
  consensus (planned) can't distinguish "old format without `run_id`"
  from "new format with `run_id` missing".
- Code: notes `extra="ignore"` as principled on `Work` specifically;
  doesn't flag the cascade to every other model. Both views together
  make the point: split `_Base` into `_OpenAlexBase(extra="ignore")`
  and `_InternalBase(extra="forbid")`.

### Theme 7 — CLI inconsistencies

- Surface: stage 3 positional args unharmonized; `--reject-file`
  legacy; `--dry-run` means different things in stage 3 (run LLM,
  discard output) and stage 5 (skip LLM, print plan); `--skip-existing`
  absent from stage 3.
- Code: `mode="x"` exclusive-create writes in stages 2, 3, 4 produce
  bare `FileExistsError` on re-run; `catalogue-fetch` help says
  `default: 199` but argparse default is `None`; `open(..., "x")` in
  screening makes one crash unrecoverable without deleting the output.
- System: I-8 (inconsistent `"x"` vs `append_jsonl` across stages),
  I-9 (stage 3's `"x"` mode blocks multi-run consensus entirely).

### Design decisions to make before stages 7–12 are planned

- Delete `_Base` — `extra="ignore"` is pydantic v2's default so the
  shared base is a no-op wrapper. Put `ConfigDict(extra="ignore")`
  directly on `Work` (and any other OpenAlex-sourced models that need
  to tolerate upstream field additions) as self-documentation. Put
  `ConfigDict(extra="forbid")` directly on every internal model
  (verdicts, meta sidecars, extraction records) so field-name typos
  fail loudly and the "old run missing `run_id`" case is explicit
  rather than silent. No new shared base class — one `ConfigDict` line
  per model.
  ([system:D-6](review-2026-04-16-system.md))
- Add `source_catalogues: list[str]` and `catalogue_ids: dict[str,
  str]` to `Work` (pr3 review theme: Clarivate / WoS enters later).
  ([system:D-3](review-2026-04-16-system.md))
- Add a derived `is_peer_reviewed: bool | None` to `Work`, computed
  at ingest from `Work.type` and `Work.primary_location.source.type`
  (OpenAlex records) or the equivalent type fields on future sources.
  Default to `None` when the source doesn't expose enough signal —
  consumers must treat missing as "unknown", not "not peer-reviewed".
  Keep the derivation in a single `derive_peer_review(work)` helper so
  it can be rerun if upstream schemas shift.
  ([system:D-4](review-2026-04-16-system.md))
- Fully-typed `ExtractionRecord` is the direction (consistent with
  "Typed data, not dicts"). The cost is that the model changes with
  the codebook during phase 3 — acceptable under "green field, no
  backwards compatibility". Add this as an explicit open question at
  the top of the (to-be-written) stage 8 doc so the decision is
  visible in the stage spec, not just in this review.
  ([system:D-13](review-2026-04-16-system.md))
- Add a quality-gate status to `ExtractedDocument` (`extraction_status`
  enum + quality metrics) so stages 7/8 can exclude garbled extractions
  rather than LLM-extracting from garbage.
  ([system:D-1](review-2026-04-16-system.md))
